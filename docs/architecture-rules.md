Шаблон 1 — как он должен выполнять любую задачу
Правило выполнения любой задачи в проекте:
Перед тем как писать код, всегда:
1. Определи, к какому домену относится задача
2. Определи, затрагивает ли задача:
   - domain model
   - database
   - API
   - UI
   - shared types
3. Проверь, не нарушает ли решение архитектурные принципы проекта
4. Только после этого вноси изменения
Если задача требует новых сущностей или новых связей, сначала обнови документацию и типы, потом реализацию.
Если задача затрагивает несколько доменов, не смешивай их в одном файле.

Шаблон 2 — шаблон для любой новой фичи
При добавлении любой новой фичи работай по шаблону:
1. Цель фичи
2. Какие домены затрагивает
3. Какие сущности используются
4. Какие таблицы/модели нужны
5. Какие API endpoints нужны
6. Какие UI экраны/компоненты нужны
7. Какие shared types нужны
8. Какие риски архитектуры могут появиться
9. Только потом реализация
Не переходи сразу к коду без этой структуры.

Шаблон 3 — запрет на архитектурную деградацию
Запрещено:
- добавлять временные костыли без явной необходимости
- дублировать сущности в разных пакетах
- смешивать DTO, database models и frontend view models в один тип
- хранить бизнес-логику внутри React-компонентов
- делать прямую зависимость frontend от database schema
- писать доступ к данным прямо внутри UI
- смешивать student и admin логику
- строить API “под один экран”
- писать структуру контента под один конкретный курс
- делать урок как одно текстовое поле, если он должен быть блочной структурой

Шаблон 4 — как проектировать сущности
При проектировании сущностей используй правила:
1. Сначала определить сущность как часть домена
2. Потом определить ее связи
3. Потом определить, что хранится в БД
4. Потом определить API representation
5. Потом определить frontend representation
Не смешивать:
- database entity
- API DTO
- UI model
Если сущность потенциально может расширяться в будущем, проектировать ее с запасом на масштабирование.

Шаблон 5 — как строить API
Правила для API:
1. Каждый домен имеет свой модуль
2. Каждый модуль имеет:
   - controller / routes
   - service
   - repository / data access
   - dto/types
3. API должен быть универсальным, не под конкретный экран
4. Endpoint должен описывать действие над сущностью, а не логику интерфейса
5. Не смешивать admin endpoints и student endpoints
6. Держать валидацию отдельно
7. Не зашивать бизнес-правила в контроллер

Шаблон 6 — как строить фронтенд mini app
Правила для frontend mini app:
1. UI не должен содержать бизнес-логику
2. Компоненты должны быть разделены на:
   - screens/pages
   - widgets
   - features
   - entities/models
   - shared/ui
3. Экран урока должен быть универсальным lesson renderer
4. LessonBlock рендерится через registry/block renderer system
5. Не делать отдельный hardcoded layout под каждый урок
6. Работа с API должна идти через отдельный слой client/services
7. Состояние пользователя, прогресса, доступа и контента не смешивать в один глобальный хаос

Шаблон 7 — как строить lesson blocks
LessonBlock system rules:
Lesson состоит из массива блоков.
Каждый блок имеет:
- id
- lessonId
- type
- orderIndex
- payload
- metadata (optional)
Поддерживаемые block types:
- text
- video
- image
- pdf
- presentation
- checklist
- quote
- download
- callout
Нужно построить систему рендера блоков через map/registry по типу блока.
Нельзя делать if-else хаос на весь экран урока.
Каждый тип блока должен рендериться отдельным компонентом.

Шаблон 8 — как строить прогресс
Правила прогресса:
1. Прогресс пользователя хранится отдельно от контента
2. Есть минимум:
   - LessonProgress
   - CourseProgress
3. Прогресс рассчитывается на backend или в отдельном domain service
4. UI только отображает прогресс, но не является источником истины
5. Завершение урока должно быть завязано на completion rules, а не на случайное локальное состояние

Шаблон 9 — как строить доступы и оплаты
Правила для доступов и оплат:
1. Payment / Purchase и Enrollment — разные сущности
2. Успешная оплата не открывает контент напрямую
3. Сначала фиксируется факт оплаты
4. Потом backend выдает доступ
5. Проверка доступа всегда идет через enrollment layer
6. Нельзя писать логику “if paid then allow” напрямую в UI или controller

Шаблон 10 — как оформлять каждый этап работы
После каждого крупного шага:
1. Кратко опиши, что было сделано
2. Какие файлы созданы/изменены
3. Какие архитектурные решения приняты
4. Какие риски предотвращены
5. Что является следующим корректным шагом
Не переходи к следующему этапу, нарушая порядок слоев архитектуры.
